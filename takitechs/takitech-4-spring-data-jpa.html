Spring Data JPA

Spring Data JPA is an sub-part of Spring Data. It is an interface that requires standards to be implemented.

It helps facilitate interactions with Hibernate, but you lose a lot of core functionalities.

How to use SDJPA

You can create entities that will be picked up by SDJPA, and do many basic CRUD operations on them, by extending your repositories with many useful interfaces, ranked from least specific to most specific:

  • Repository
    • does not define anything, is simply a marker
  • CrudRepository
    • only defines create, read, update, delete
  • PagingAndSortingRepository, ListPagingAndSortingRepository
    • helps with pagination and sorting

Specifications

In Spring data JPA, a specification is a tool to make more complex and tailored queries, with and and or clauses. It helps writing dynamic queries that you can combine, match and mix.

Why do you need specifications ?

When you have a lot of clauses in your search query, you will end up with very long names. Specifications will be useful to add specific, tailored clauses and have easier to read names.

@NameQuery and @Query

Before, you'd put @NameQuery on an entity to define a specific query defined for a field in your entity.
Query is put on a function to link the function to SDJPA. However, this function is static, and you'd have to write a new one for each new query you want to define.

Spring Data

Spring data is a spring project that helps you interact with data sources.

Predicates

They're used as individual clauses that you can mix and match with or and and links between them to make a dynamic and custom query.

Projections

They're views of an entity where you choose the information sent by cutting the fields sent. You use DTOs to achieve this.

Hibernate

Hibernate is an implementation of Spring data JPA, and is considered an ORM.

Another implementation is EclipseLink.

When to use Hibernate ?

You'd use hibernate when:

  • You don't have too complex queries.

You don't want to use hibernate when:

  • You have many complex computations to do in your back-end
    You'd use JDBC and JOOQ if you really wanted to do heavy computation.

Why AOP ?

The technical side of the code does not generate value, which is why we want to hide it.

AOP

AOP means aspect oriented programming It's a paradigm that separates the technical side of your code with the functional side.

How to use Aspects ?

There are different ways of using aspects:

  • Annotations
  • Target a method by its name with advices, pointcuts and joinpoints:
    • Joinpoints are where the aspect will be used, all the methods annotated with the aspect.
    • Pointcuts are the predicates, it defines who are the joinpoints.
    • Advices are when the aspects will trigger. The most important one is around.

Why use annotations ?

You want to use annotations mainly because it's more visible what triggers when.

Weaving

Weaving is when you link your technical code with your functional code. Spring will use Run time to weave aspects with your code. Here are all the different steps of code running:

  • Compile time is when our code compiles our java code into bytecode.
  • Load time is when the bytecode is loaded in the JVM
  • Run time is when the code is running.

@Retention

This annotation is used to decide when to keep or get rid of an aspect:

  • Source is only kept when the code is not compiled.
  • Class is only kept in the bytecode
  • Runtime is only kept when the application is run
