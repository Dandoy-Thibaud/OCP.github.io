DevOps

DevOps is a way of coupling the code with the deployment environment, and helps shorten the development time of an application by implementing tools and good practices.

Tools

There are many tools to help with DevOps automatization:

  • Kubernetes
  • GitLab CI
  • Docker
  • Kube
  • etc...

Pipelines

An ensemble of steps that code is supposed to go through to check if that code is correct on a technical and quality side.

Why ?

It reduces human error because it automatises tests and suggests code amelioration.

SonarLint

SonarLint helps linting, check test coverage, and check for easy-to-miss errors regarding code robustness and security, allowing you to think about more important aspects of your code.

Docker

Docker helps by creating a container to run an application. Docker doesn't create VMs, as it uses the host's ES with images.
You use layers with your dockerfile so that Docker is able to save every subsequent one of them to its cache so that it doesn't have to build each step again.

Why do we DevOps ?

DevOps helps define responsibilities by showing developers how their code is meant to run in production, and showing Ops how to communicate with Devs. It favors continuous development.

Orchestrator

An orchestrator can do a lot, such as providing

  • a host of dynamic services for running,
  • connecting,
  • scaling,
  • managing complex multi-container workloads.

Argo

Argo pulls regularly pipelines to show their status without having to go check.

Continuous Integration & Deployment

Continuous Integration is the pipeline going through building, testing and packaging.
Continuous Deployment also adds deployment to production.

Git

We use github flow at Takima, meaning that we create a branch per feature, before merging it fast-forward onto main after rebasing it if possible.
Of course, we use code reviews, QA, and we make sure that the main branch is deployable at any moment.

Hotfixes

Hotfixes require that a new branch be created for a bug to be fixed. They are urgent fixes, meaning that they will be merged with the deployed branch as soon as possible.
Furthermore, while you could fix directly on the main, you don't do that as you risk regressing that branch.
You may also rollback to a previous version if a hotfix is not advisable.

Production

Logs

In production, we use a tool such as sentry to log every error and report it somewhere. There is also a designated firefighter that will investigate every reported bug
You may also use datadog that will record user actions to pinpoint what the problem is, as well as enabling notifications so that you don't check the production server all day.
